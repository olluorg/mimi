# Event System

## Definition

An `Event` is an immutable record of something that happened in the interaction layer.

Events are:
- produced by the input system (touch, mouse, keyboard)
- dispatched by the engine to behaviors
- never produced by behaviors themselves
- never mutated after creation

## Event Structure

```
Event {
    type: EventType
    tick: Tick
    source: EntityId?      // entity that originated the event (nullable for system events)
    target: EntityId?      // entity that received the event (nullable)
    payload: EventPayload  // type-specific data
}
```

## Event Types

### Input Events

| Event | Trigger | Payload |
|---|---|---|
| `Tap` | Single touch release | `position: Vec2` |
| `DragStart` | Touch hold + move begins | `position: Vec2`, `entityId: EntityId` |
| `DragMove` | Touch moves while dragging | `position: Vec2`, `delta: Vec2` |
| `DragEnd` | Touch released after drag | `position: Vec2`, `entityId: EntityId` |
| `Drop` | Dragged entity released over target | `source: EntityId`, `target: EntityId` |
| `Hold` | Touch held for duration threshold | `position: Vec2`, `duration: Tick` |
| `TracePoint` | Touch moves along trace path | `position: Vec2`, `progress: Float` |
| `TraceComplete` | Trace path finished | `pathId: String`, `accuracy: Float` |

### System Events

| Event | Trigger | Payload |
|---|---|---|
| `MatchSuccess` | Constraint match confirmed | `source: EntityId`, `target: EntityId` |
| `MatchFail` | Constraint match rejected | `source: EntityId`, `target: EntityId`, `reason: String` |
| `ObjectiveProgressed` | Objective moved to Progressing | `objectiveId: String` |
| `ObjectiveCompleted` | Objective moved to TentativeComplete | `objectiveId: String` |
| `SceneCompleted` | All objectives LockedComplete | — |
| `HintTriggered` | Adaptive hint behavior fired | `entityId: EntityId`, `hintType: String` |

## Dispatch Rules

1. Events are dispatched to behaviors registered for that event type.
2. All eligible behaviors receive the event in the same tick.
3. A behavior may produce zero or more `Command` objects per event.
4. System events (`MatchSuccess`, `ObjectiveCompleted`, etc.) are generated by the engine after World commit — they may trigger behaviors in the **next** tick.

## Event Ordering

Within a single tick, multiple events may be queued (e.g. `DragMove` + `Drop`).

Rules:
- Events are processed **sequentially**, one full pipeline per event
- Two events never share a tick (tick increments between events)
- Event queue is drained in arrival order (FIFO)

## Invariants

1. Behaviors never produce events directly.
2. Events are never modified after dispatch.
3. System events are always generated **after** World commit, never during.
4. `event.tick` always equals the engine's current `tick` at dispatch time.
